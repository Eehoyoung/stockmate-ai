## 1. 시스템 전체 아키텍처 개요

```
[Kiwoom WebSocket]
       │ 실시간 체결/호가/VI
       ▼
[websocket-listener] ──────────────────────────────────────────────────────┐
   • ws:tick:{stk_cd}     (hash, 체결 데이터)                              │
   • ws:hoga:{stk_cd}     (hash, 호가 데이터)                              │
   • ws:expected:{stk_cd} (hash, 동시호가 예상체결가)                       │ Redis
   • ws:strength:{stk_cd} (list, 체결강도 시계열)                           │
   • vi_watch_queue       (list, VI 이벤트)                                │
   • vi:{stk_cd}          (hash, VI 발동 내역)                             │
       │                                                                   │
[api-orchestrator] ──────────────────────────────────────────────────────▶ │
   TradingScheduler                                                        │
   CandidateService          candidates:s{N}:{market} (list) ─────────────┘
   StrategyService                                                         │
       │                                                                   │
[ai-engine (Python)]  ◀────────────────────────────────────────────────────┘
   candidates_builder.py  ← 후보 풀 보조 생성
   strategy_runner.py     ← 전략별 스캔
   queue_worker.py        ← 파이프라인 오케스트레이션
   scorer.py              ← 규칙 기반 점수화
   analyzer.py            ← 전처리/필터
   confirm_worker.py      ← Claude API 2차 분석
       │
       │ LPUSH telegram_queue  (1차 신호)
       │ LPUSH ai_scored_queue (최종 신호)
       ▼
[telegram-bot (Node.js)]
   signals.js  ← ai_scored_queue RPOP
   formatter.js ← 메시지 포매팅 (슬리피지/실질R:R 포함)
       │
       ▼
[Telegram 채널/그룹]
```

## 4. 신호 파이프라인 상세 (queue_worker.py)

```
[telegram_queue] ← RPOP (2초 폴링)
       │
       ├─ 1. news:trading_control == "PAUSE"?  → action=CANCEL, LPUSH ai_scored_queue
       │
       ├─ 2. ws:py_heartbeat 확인 (WS 오프라인 → 경고 로그, 계속 진행)
       │
       ├─ 3. _build_market_ctx()
       │      ├─ ws:tick:{stk_cd}     → flu_rt, cur_prc, vol
       │      ├─ ws:hoga:{stk_cd}     → bid_ratio, ask_ratio
       │      ├─ ws:strength:{stk_cd} → cntr_str (체결강도)
       │      └─ vi:{stk_cd}          → vi_price, is_dynamic (S2 전용)
       │
       ├─ 4. rule_score(signal, ctx) → (float, components dict)
       │      scorer.py CLAUDE_THRESHOLDS 참조
       │
       ├─ 5. should_skip_ai(r_score, strategy)
       │      r_score < CLAUDE_THRESHOLDS[strategy] → action=CANCEL
       │
       ├─ 6. R:R 필터 (MIN_RR_RATIO = 1.3)
       │      (tp1 - cur) / (cur - sl) < 1.3 → action=CANCEL
       │
       ├─ 7. check_daily_limit(rdb)
       │      claude:daily_calls:{YYYYMMDD} >= MAX_DAILY_CLAUDE → action=CANCEL
       │
       ├─ 8. analyze_signal()  →  confirm_worker.py
       │      모델: claude-sonnet-4-6
       │      MAX_TOKENS: 512
       │      TIMEOUT: 10s
       │      결과: action=BUY|HOLD|CANCEL + ai_score + reasoning
       │
       ├─ 9. LPUSH ai_scored_queue (최종 신호)
       │
       └─ 10. PostgreSQL
              update_signal_score()
              insert_score_components()
```

## 8. Kiwoom API 오류 처리 공통 규칙

Kiwoom REST API는 **서버 내부 오류도 HTTP 200으로 반환**한다. `raise_for_status()` 감지 불가.

| 구분 | 판단 방법 |
|------|----------|
| 정상 | `return_code == "0"` |
| API 오류 | `return_code != "0"` (예: `"-1"`, `"1700"`) |
| 서버 내부 오류 | `"error"` 키 존재 (`{"error":"INTERNAL_SERVER_ERROR","status":500,...}`) |

모든 전략 파일에서 `http_utils.validate_kiwoom_response(data, api_id)` 호출 필수.
