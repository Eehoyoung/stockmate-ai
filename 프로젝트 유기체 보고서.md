## 2. 모듈별 역할

### 2-1. `websocket-listener/` (Python asyncio)

| 파일 | 역할 |
|------|------|
| `main.py` | 진입점, Redis 연결, ws_client 태스크 기동 |
| `ws_client.py` | Kiwoom WebSocket 연결·LOGIN·구독·재연결, GRP 1~4 할당 (JAVA_WS_ENABLED=false 시 단독) |
| `logger.py` | JSON 구조화 로거 (JSON Lines 형식, KST 타임스탬프, service/module/request_id 필드) |
| `redis_writer.py` | 수신 데이터를 Redis 키에 기록 (tick·hoga·strength·vi·heartbeat) |
| `token_loader.py` | Redis 에서 Kiwoom Bearer 토큰 로드 |
| `health_server.py` | HTTP `/health` 엔드포인트 (기본 포트 8081) |

- `JAVA_WS_ENABLED=false` (기본) 상태에서 Python이 GRP 1~4 단독 사용; Java WS 활성화 시 Java GRP 1~4, Python GRP 5~8 분리
- 장 운영 시간 (평일 07:30~15:35 KST) 외 자동 대기, `BYPASS_MARKET_HOURS=true` 로 우회 (기본값 `true` – 테스트 편의)
- WS 연결 시 `authorization` Bearer 헤더 제거 완료 (cb96c07); LOGIN 메시지 내 `return_code` 필드만 사용
- `tests/` 디렉토리: `test_failure_reproduction.py` (토큰 만료·네트워크 단절·파싱 오류), `test_mttr.py` (60초 이내 복구 검증), `test_reconnect_scenarios.py` (재연결 시나리오)

## 4. 데이터 흐름 상세

### 4-1. 정규 매매 신호 흐름 (기존 CANCEL 경로)

```
TradingScheduler (크론)
  → StrategyService.scanXxx()
  → SignalService.processSignal(dto)
      ├─ 중복 체크 (signal:dup:)
      ├─ 종목 쿨다운 (signal:stock_cooldown:)
      ├─ 일별 상한 (signal:daily_count:)
      └─ LPUSH telegram_queue

ai-engine queue_worker
  → RPOP telegram_queue
  → 뉴스 PAUSE 여부 확인 (news:trading_control)
  → _build_market_ctx() : ws:tick·ws:hoga·ws:strength·vi 읽기
  → rule_score() : 전략별 규칙 스코어 계산
  → should_skip_ai() : 임계값 미달?
      ├─ YES → result = CANCEL → LPUSH ai_scored_queue
      └─ NO  → LPUSH human_confirm_queue (Human Confirm Gate)

telegram-bot signals.js
  → RPOP ai_scored_queue
  → action=CANCEL → 무시
```

### 4-2. Human Confirm Gate 흐름 (신규)

```
ai-engine queue_worker
  → rule_score >= threshold
  → LPUSH human_confirm_queue {item + rule_score + market_ctx}
  → SETEX confirm_pending:{sig_id} 1800s payload

telegram-bot signals.js startConfirmPoller
  → RPOP human_confirm_queue
  → SET confirm_pending:{sig_id} EX 1800  (중복 보장)
  → sendMessage(chatId, text, inline_keyboard)
      ┌─ ✅ 분석 진행 → callback_data: confirm_yes:{sigId}
      └─ ❌ 취소      → callback_data: confirm_no:{sigId}

사용자가 버튼 클릭
  ├─ confirm_yes:{sigId}
  │    → GET confirm_pending:{sigId}
  │    → LPUSH confirmed_queue payload
  │    → DEL confirm_pending:{sigId}
  │    → editMessageText "분석 진행 중"
  └─ confirm_no:{sigId}
       → DEL confirm_pending:{sigId}
       → editMessageText "신호 취소"

ai-engine confirm_worker
  → RPOP confirmed_queue
  → check_daily_limit()
  → analyze_signal() (Claude API)
  → enriched.pop("market_ctx")   ← 대용량 데이터 제거
  → LPUSH ai_scored_queue {enriched + human_confirmed=True}
```

### 4-3. VI 눌림목 신호 흐름 (S2)

```
websocket-listener ws_client.py
  → GRP 7 (1h) VI 발동·해제 수신
  → write_vi() → HSET vi:{stk_cd}
  → LPUSH vi_watch_queue {stk_cd, vi_type, ...}

api-orchestrator ViWatchService
  → RPOP vi_watch_queue
  → StrategyService.checkViPullback()
  → SignalService.processSignal()
  → LPUSH telegram_queue
  → (이후 ai-engine 흐름과 동일)
```

### 4-4. 강제청산·일별 리포트 (특수 타입)

```
api-orchestrator
  → LPUSH telegram_queue {type: "FORCE_CLOSE" | "DAILY_REPORT", ...}

ai-engine queue_worker
  → type in ("FORCE_CLOSE", "DAILY_REPORT")
  → LPUSH ai_scored_queue 즉시 통과 (AI 분석 없음)

telegram-bot signals.js
  → FORCE_CLOSE → formatForceClose() → sendMessage
  → DAILY_REPORT → formatDailyReportEnhanced() → sendMessage
```

### 4-5. 뉴스 매매 제어 흐름

```
ai-engine news_scheduler (30분 주기)
  → 뉴스 API 조회 → Claude 분석
  → SET news:trading_control (CONTINUE|CAUTIOUS|PAUSE)
  → 변경 시 LPUSH ai_scored_queue {type: "NEWS_ALERT"}
  → PAUSE 시 LPUSH ai_scored_queue {type: "PAUSE_CONFIRM_REQUEST"}

telegram-bot signals.js
  → NEWS_ALERT → sendMessage (알림)
  → PAUSE_CONFIRM_REQUEST → sendMessage (confirm_pause / cancel_pause 버튼)

사용자 confirm_pause 클릭
  → kiwoom.setTradingControl('PAUSE')
  → POST /api/trading/control/PAUSE
  → Redis news:trading_control = PAUSE

ai-engine queue_worker (신호 처리 시)
  → GET news:trading_control == "PAUSE"
  → result = CANCEL (모든 신호 취소)
```

## 5. Human Confirm Gate 흐름 (세부)

이번 세션에서 추가된 핵심 기능. 기존에는 규칙 점수 통과 즉시 Claude API를 호출했으나, 이제는 반드시 사람의 승인이 있어야 Claude 분석이 진행된다.

```
                  ┌───────────────────────────────────┐
                  │     queue_worker (ai-engine)      │
                  │                                   │
  telegram_queue  │  score < threshold                │
  ──────────────► │  ────────────►  ai_scored_queue   │
                  │  (CANCEL)                         │
                  │                                   │
                  │  score >= threshold               │
                  │  ────────────►  human_confirm_queue│
                  └───────────────────────────────────┘
                                       │
                  ┌────────────────────▼──────────────┐
                  │  startConfirmPoller (telegram-bot) │
                  │                                   │
                  │  RPOP human_confirm_queue          │
                  │  sendMessage + InlineKeyboard      │
                  │    ✅ confirm_yes:{sigId}          │
                  │    ❌ confirm_no:{sigId}           │
                  └───────────────────────────────────┘
                                       │
                  ┌────────────────────▼──────────────┐
                  │    사용자 버튼 응답 (callback_query) │
                  │                                   │
                  │  confirm_yes → LPUSH confirmed_q  │
                  │  confirm_no  → DEL pending key    │
                  └───────────────────────────────────┘
                                       │
                  ┌────────────────────▼──────────────┐
                  │   confirm_worker (ai-engine)      │
                  │                                   │
                  │  RPOP confirmed_queue             │
                  │  check_daily_limit()              │
                  │  analyze_signal() → Claude API    │
                  │  pop("market_ctx")                │
                  │  LPUSH ai_scored_queue            │
                  │    human_confirmed = True         │
                  └───────────────────────────────────┘
```

**타임아웃**: 컨펌 대기 시간은 1800초(30분). 이 시간 내 응답 없으면 `confirm_pending:{sig_id}` TTL 만료로 자동 무효화.

```json

  "strategy": "S1_GAP_OPEN",
  "action": "ENTER",
  "message": "[S1_GAP_OPEN] 삼성전자 갭 4.2% ...",
```

```json

  ...telegram_queue 원본 필드,
```

## 9. 알려진 제약사항 및 주의점

1. **`confirm_pending` 이중 저장**: `push_human_confirm_queue` 에서 SETEX로 저장하고, `sendConfirmRequest` 에서도 SET EX 한다. 두 번째 SET이 첫 번째를 덮어쓰는데 `market_ctx` 포함 여부가 동일하므로 문제없다. 단, TTL이 sendConfirmRequest 시점부터 재기산됨.

2. **`confirmed_queue` 방향**: ai-engine은 LPUSH → RPOP 순서로 FIFO를 보장한다. telegram-bot의 confirm_yes 핸들러도 `redis.lpush('confirmed_queue', ...)` 방식이므로 일관성 있음.

3. **`market_ctx` 생략 가능성**: `confirmed_queue` 에는 market_ctx 가 포함된 상태로 들어오나, confirm_worker 에서 pop 후 제거하여 ai_scored_queue 에는 포함되지 않는다. Claude 분석 시에는 `ctx = item.get("market_ctx", {})` 로 추출하여 사용한다.

4. **`sig_id` 충돌 위험**: `id` 필드가 없으면 `stk_cd` 를 사용하므로, 동일 종목에서 여러 전략 신호가 동시에 발생하면 `confirm_pending:{stk_cd}` 가 덮어쓰여질 수 있다. id 필드를 항상 포함하도록 api-orchestrator에서 보장하는 것이 권장된다.

5. **`ENABLE_CONFIRM_GATE` 환경변수 없음**: confirm_worker는 항상 기동된다. 컨펌 게이트를 비활성화하려면 현재 코드 수정이 필요하다.

6. **`hibernate.ddl-auto: create`**: application.yml 주의 – 프로덕션 DB에서 실행 시 테이블 재생성 위험.

7. ~~**`BYPASS_MARKET_HOURS` 기본값 `true`**~~ → **해소됨**: `ws_client.py` 에서 하드코딩 제거, `os.getenv("BYPASS_MARKET_HOURS", "false")` 로 복원. 기본값은 `false`.

8. ~~**`ma_utils.py` API 의존**~~ → **해소됨**: `fetch_daily_candles` 에 인메모리 TTL 캐시(`_CANDLE_CACHE`, 기본 3600s) 추가. 동일 종목 중복 API 호출 차단. TTL은 `MA_CACHE_TTL_SEC` 환경변수로 조정 가능.

9. ~~**GRP 할당 충돌 주의**~~ → **해소됨**: `run_ws_loop` 시작 시 `JAVA_WS_ENABLED=true` 감지하면 ERROR 레벨 로그로 충돌 위험 경고 출력. 운영자가 즉시 인지 가능.
