# StockMate AI P0/P1/P2 작업계획서 초안

## 문서 목적
- 최근 점검 결과를 기준으로 기능 완성도와 운영 완성도를 함께 끌어올리기 위한 실행 우선순위를 정리한다.
- 본 문서는 시니어 PM 관점에서 작성한 초안이며, PM 메모와 퀀트 관점 요구사항을 통합한 계획서다.

## P0
### 목적
- 배포 직후 장애를 줄이고, 신호 파이프라인의 병목 지점을 매일 확인 가능한 상태로 만든다.
- "돌아간다" 수준이 아니라 "운영 가능하다" 수준의 기준선을 세운다.

### 핵심 작업
- 배포 설정 검증 게이트 도입: `telegram-bot`, `api-orchestrator`, `websocket-listener`의 필수 env, KST 시간대, Redis/Postgres 연결값을 기동 전에 자동 검증한다.
- 신호 파이프라인 일일 헬스체크 정의: `후보 생성수 -> 규칙 통과수 -> AI 통과수 -> Telegram 발송수 -> 최종 성과`를 전략별로 집계한다.
- 신호 품질 기준선 수립: 전략별 최소 샘플 수, 허용 승률, 손익비, timeout 허용치, 최대 CANCEL 비율을 명문화한다.
- 운영자용 기본 명령 체계 정리: `/status`, `/signals`, `/today`, `/health`의 응답 포맷과 필수 정보를 표준화한다.

### 완료 기준 (DoD)
- 세 서비스 모두 필수 설정 누락 시 기동 전에 명확한 에러와 함께 실패한다.
- 일일 집계 리포트에서 전략별 파이프라인 수치가 누락 없이 확인된다.
- 운영자가 Telegram만으로 당일 상태와 신호 수를 확인할 수 있다.
- 품질 기준선 문서가 작성되고, 최소 1일치 실데이터로 기준선 리포트가 생성된다.

### 미조치 리스크
- 환경변수 누락 한 번으로 봇 중단, WS 미연결, 스케줄 오작동이 반복된다.
- 신호가 적거나 이상할 때 로그 추적에만 의존하게 되어 운영 피로가 커진다.
- 규칙 점수와 실제 발송 수가 어긋나도 합의된 기준 없이 판단이 흔들린다.
- 운영 메시지와 실제 장애 징후를 빠르게 구분하지 못한다.

## P1
### 목적
- 장전~장초 신호 품질을 안정화하고, 전략별 동작 시간과 timeout 정책을 현실에 맞게 재설계한다.
- 코드/데이터 정합성을 높여 동일 종목과 동일 이벤트를 일관되게 추적할 수 있게 만든다.

### 핵심 작업
- 후보풀 안정화: 후보 생성 시점을 `07:50~08:20 KST`로 앞당기고, `08:25 KST` 기준 후보풀 스냅샷을 고정한다.
- 전략 시간창 재정의: 장전형 `S1,S7`, 장초형 `S3,S6,S11`, 종가형 `S12`를 분리하고, 각 전략별 실행 가능 시간대를 명시한다.
- 전략별 timeout 원인 분해: `데이터 대기`, `외부 API`, `연산 병목`, `큐 적체`로 분류해 로그 스키마를 표준화한다.
- 점수-발송 정책 재보정: `S8,S9,S10,S12` 등 고득점 대비 CANCEL 비율이 높은 전략의 임계값과 변환표를 재조정한다.
- 종목코드 canonical 정책 완료: Redis 키, DB, Telegram 메시지, 리포트 전부 `정규 코드만 허용`으로 강제한다.
- WS 세션 정책 문서화: `KST 월-금 07:30~20:10` 연결 원칙과 예외 조건을 운영 규칙으로 고정한다.

### 완료 기준 (DoD)
- 장전 구간에 후보풀 fallback 발생률이 유의미하게 감소한다.
- 전략별 timeout 로그가 원인 분류와 경과 시간까지 포함한다.
- 고득점 대비 CANCEL 과다 전략에 대해 보정 결과와 근거가 문서화된다.
- `_AL` 등 비정규 접미사가 신규 Redis/DB/메시지 경로에서 더 이상 생성되지 않는다.
- WS 연결 정책이 코드, 설정, 문서에서 동일하게 표현된다.

### 미조치 리스크
- 장초 핵심 전략이 매일 다른 입력 상태로 돌아 재현성과 성과 해석이 무너진다.
- 느린 전략이 계속 timeout으로 누락되는데 원인을 특정하지 못한다.
- 높은 규칙점수를 받아도 실제로는 거의 발송되지 않는 왜곡이 지속된다.
- 동일 종목이 여러 코드 형태로 남아 분석, 캐시, 통계가 오염된다.
- 장외 연결 범위에 대한 팀 해석이 갈려 불필요한 노이즈가 반복된다.

## P2
### 목적
- 신호 품질을 사용자 체감 기준과 사후평가 기준으로 끌어올리고, 운영 학습 루프를 고정한다.
- 기능 추가보다 "같은 품질로 반복 운영되는 서비스" 상태를 만드는 데 집중한다.

### 핵심 작업
- 유동성 필터 강화: 장전~장초 신호에 예상 체결량, 호가 스프레드, 동시호가 참여도를 반영한다.
- 동일 종목 재발송 정책 정리: 같은 전략, 다른 전략, 재기동 후 중복 발송을 구분해 처리한다.
- 사후평가 표준화: `5분`, `15분`, `종가` 기준 MFE/MAE와 손익비를 전략별로 추적한다.
- 과정 품질 중심 모니터링 전환: 후보 수, 평가 수, timeout 수, cancel 사유 비중, WS 지연을 핵심 운영 지표로 삼는다.
- 전략 warm-up 절차 도입: `07:20~08:20 KST` 사이 Redis, 후보풀, 종목명 캐시, WS heartbeat를 사전 점검한다.
- 주간 회고와 변경관리 체크리스트 운영: 전략 성과, 누락 사례, 운영 장애, 사용자 피드백을 주간 단위로 검토한다.

### 완료 기준 (DoD)
- 초보자 기준으로 체결 가능한 신호 비중이 높아졌다는 근거 지표가 확보된다.
- 동일 종목 중복 발송 정책이 문서와 코드로 일치한다.
- 전략별 사후평가 리포트가 자동 생성되고, 최소 1주 단위 비교가 가능하다.
- 장중 운영 대시보드에서 신호 수뿐 아니라 과정 품질 지표를 확인할 수 있다.
- 주간 리뷰와 변경 체크리스트가 실제 운영 루틴에 포함된다.

### 미조치 리스크
- 백테스트상 좋아 보여도 실전 체결이 어려운 신호가 계속 섞인다.
- 재기동 후 중복 알림과 성과 왜곡이 반복된다.
- 적중률만 보고 전략을 유지하거나 제거하는 잘못된 판단이 누적된다.
- 로그를 수작업으로 뒤져야 해서 운영 학습 속도가 느려진다.
- 변경은 계속되는데 무엇이 실제 개선인지 판단하기 어려워진다.

## 우선순위 요약
- P0는 "배포 실패 방지와 운영 가시성 확보"다.
- P1은 "장전/장초 신호 품질 안정화와 정책 정합성 확보"다.
- P2는 "사용자 체감 품질과 반복 운영 학습 체계 고도화"다.
