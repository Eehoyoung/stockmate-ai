## 1. 공통 후보 풀 (candidates:{market})

**핵심 단일 소스 – `CandidateService.getCandidates()`**

| 항목 | 내용 |
|------|------|
| API | `ka10029` 예상체결등락률상위요청 |
| 필터 | 갭등락률 **3% ~ 30%**, 만주 이상, 관리종목 제외, 1천원 이상 |
| 정렬 | 상승률 순 (`sort_tp=1`) |
| 캐시 | Redis `candidates:001`, `candidates:101` / TTL **3분** |
| 한도 | 최대 200개 / 시장 |
| 호출 시점 | 각 전략 스캐너가 캐시 없을 때 자동 갱신 |

### 문제점
- `ka10029`는 **장전 예상체결 데이터**가 주 원천 → 장중에는 "장전 갭 예상치" 기준이므로 실시간 장중 모멘텀을 반영하지 못함
- 전략 특성(수급형/기술형/과매도형)과 무관하게 **모든 전략이 동일한 갭 상위 풀** 사용
- 갭 3% 기준이 너무 낮아 S14(과매도 반등)처럼 **하락 중인 종목**은 이 풀에 아예 없음
- 스윙 전략(S8/S9/S13/S14/S15)은 전일 종가 기준 분석이 필요하지만, 당일 갭 기준으로 후보가 왜곡됨

## 5. 고도화 방향 제안

```
현재 (단일 풀):           개선 후 (목적별 3개 풀):
  ka10029 갭3~30%  →     [갭 풀]   ka10029 갭2~20%  (S1, S7)
```

### 우선순위
1. **즉시** – S14 전용 과매도 후보 풀 (`ka10027` 하락 방향)
2. **단기** – S9/S13 스윙 전용 후보 풀 (`ka10027` 소폭 상승 + `ka10031` 전일 거래량)
3. **단기** – Java S10 → `ka10016` 교체, Java S11 → `ka10035` 교체
4. **중기** – 후보 풀 분리 아키텍처 (`CandidateService` 리팩토링)
