## G2. 슬리피지·수수료 미반영 [P1]

### 현황

현재 전략별 목표가·손절가는 명목 % 기준이다. 실제 매수·매도 체결 시:

| 비용 항목 | 금액 | 비고 |
|---------|------|------|
| 키움 수수료 (매수+매도) | 약 0.03% | 위탁매매 기준 |
| 세금 (매도 시) | 0.20% (코스닥), 0.18% (코스피) | 증권거래세 |
| 시장가 슬리피지 | 0.1~0.5% | 코스닥 중소형주 최대 0.5% |
| **합계 실비** | **0.33~0.73%** | 매수·매도 왕복 기준 |

**예시 영향 (S1, 목표 +4.0%, 손절 -2.0%)**:
```
명목 R:R = 4.0 / 2.0 = 1:2.0
실질 R:R = (4.0 - 0.5) / (2.0 + 0.5) = 3.5 / 2.5 = 1:1.4  ← 30% 열화
```

### 해결 방안: 실비 반영 유효 R:R 표시 추가

전략 신호 메시지와 성과 추적에 `effective_rr` 필드를 추가하여 사용자에게 실질 R:R을 제공한다. 목표가·손절가 자체는 변경하지 않으며(시장 기술적 기준 유지), 별도 필드로 안내한다.

#### 구현 위치 및 변경 내용

**`ai-engine/analyzer.py`**:
```
SLIP_FEE = {
    "KOSPI": 0.0035,   # 수수료 0.03% + 세금 0.18% + 슬리피지 0.15%
    "KOSDAQ": 0.0045,  # 수수료 0.03% + 세금 0.20% + 슬리피지 0.25%
}
→ effective_target = target_pct - slip_fee
→ effective_stop   = stop_pct   - slip_fee  (손절도 슬리피지 반영)
→ effective_rr     = effective_target / abs(effective_stop)
```

**`telegram-bot/src/utils/formatter.js`**:
- 신호 메시지에 `실질 R:R: {effective_rr}` 라인 추가
- effective_rr < 1.0 이면 ⚠️ 표시

**신규 API 필요 없음** — 내부 계산 로직 추가만으로 구현 가능

## G3. 오버나잇 갭다운 리스크 미관리 [P1]

### 현황

스윙 전략(S8~S15)은 당일 종가 매수 후 3~10거래일 보유를 권고한다. 보유 기간 중 악재 발생 시:

- **갭다운**: 전일 종가 대비 -3~10% 하락 시가 → 손절선 이미 이탈 후 개장
- **현재 대응**: 없음. 사용자가 직접 HTS 모니터링 필요

### 해결 방안: 장전 갭다운 조기 경보 알림

장전 동시호가 시간(08:00~09:00)에 현재 오버나잇 보유 신호(SENT 상태)의 예상 시초가를 체크하여 손절선 근접·이탈 시 텔레그램 알림을 발송한다.

```
흐름:
  08:30 스케줄러 실행
    → PostgreSQL에서 SENT 상태 스윙 신호 목록 조회
    → 각 종목의 예상 시초가 조회 (Redis ws:expected OR ka10087)
    → 예상가 ≤ 손절가 × 1.01 이면 OVERNIGHT_RISK_ALERT 발행
    → ai_scored_queue → telegram-bot → 사용자 알림
```

#### 사용 API

| API ID | API명 | 역할 | 호출 시점 |
|--------|-------|------|---------|
| **0H** (WebSocket) | 주식예상체결 | 장전 예상 시초가 실시간 수신 | 이미 구독 중 (ws:expected 키) |
| **ka10087** | 시간외단일가요청 | 장전 시간외 단일가 확인 (WS 데이터 보완) | 08:30 fallback |
| **ka10029** | 예상체결등락률상위요청 | 갭다운 상위 종목 빠른 스캔 | 08:20 선제 확인 |

#### 알림 메시지 형식

```
⚠️ [오버나잇 리스크 경보]
005930 삼성전자 — S8_GOLDEN_CROSS
진입가: 84,300원  손절가: 80,928원 (-4.0%)
예상 시초가: 81,200원 (전일비 -3.7%)
→ 손절선(80,928원)까지 잔여: +272원 (+0.3%)

손절 주문 미리 확인하세요.
```

#### 구현 위치

- `api-orchestrator/scheduler/OvernightRiskScheduler.java` 신규
  - `@Scheduled(cron="0 30 8 * * MON-FRI")` – 08:30 실행
  - `TradingSignalRepository.findSentSwingSignals()` – SENT + S8~S15 신호 조회
  - `RedisMarketDataService.getExpectedPrice()` – ws:expected 조회, 없으면 ka10087 fallback
  - 손절 -1% 이내(근접) → WARNING / 손절 이탈 → DANGER 레벨 분리 발행

## G6. Claude 점수 일관성 검증 [P2]

### 현황

확률적 LLM 특성상 동일 신호를 복수 전송하면 ai_score가 ±5~10점 편차 가능. 65점 경계 근방 신호(62~68점)는 통과 여부가 일관되지 않을 수 있다.

### 해결 방안: 경계 신호 재확인 로직

임계값 ±5 이내 신호를 2회 Claude에 재전송하여 평균 점수로 판정한다.

```python
# analyzer.py 변경안
if abs(ai_score - threshold) <= 5:
    # 경계 구간 재확인: 1회 추가 호출
    second_result = await _call_claude(signal_data)
    ai_score = (ai_score + second_result["ai_score"]) / 2
```

**비용 영향**: 하루 신호 30건 중 경계 구간 비율을 20%로 가정하면 최대 +6건/일 Claude 추가 호출. `MAX_CLAUDE_CALLS_PER_DAY` 100 대비 영향 미미.

**신규 API 필요 없음**
