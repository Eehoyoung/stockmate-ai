### [C-3] `hibernate.ddl-auto: update` + Flyway 병행 — 스키마 일관성 파괴

**관련 파일**
- `api-orchestrator/src/main/resources/application.yml:8,11–16`

**현상**

```yaml
jpa:
  hibernate.ddl-auto: update      # Hibernate가 엔티티 기반 DDL 자동 수정
flyway:
  enabled: true
  validate-on-migrate: false      # 충돌 검증 비활성화
```

두 개의 DDL 엔진이 동시에 스키마를 제어하는 상태.

**DB 분석**
- Hibernate `update`는 `flyway_schema_history`에 기록을 남기지 않아 로컬·staging·prod 스키마가 점점 diverge된다.
- `validate-on-migrate: false`가 이 충돌을 감지조차 못하게 막는다.
- 장기적으로 Flyway 마이그레이션 실패 위험 증가.

**수정 방향**

```yaml
jpa:
  hibernate.ddl-auto: validate   # Flyway가 DDL 소유, Hibernate는 검증만

flyway:
  enabled: true
  validate-on-migrate: true      # 스키마 불일치 시 앱 기동 차단
  baseline-on-migrate: true
  baseline-version: 1
```

신규 컬럼·테이블 추가 시 반드시 `db/migration/V{N}__{설명}.sql` 작성 필수.

### [H-4] `skip_entry` R:R 필터 미연결 — 실질적 무기능

**관련 파일**
- `ai-engine/queue_worker.py:131–144`

**현상**

```python
# queue_worker.py:131-144
if signal.get("skip_entry"):
    rr_val = signal.get("rr_ratio", 0.0)
    action = "CANCEL"
else:
    action = "ENTER"   # skip_entry가 False면 항상 여기
```

`skip_entry=True`는 `tp_sl_engine.py`가 R:R < 1.3 이면 설정해야 한다. 그러나 `queue_worker.py` 내에 `tp_sl_engine` 호출이 없다.  
Java `TradingSignal.skipEntry` 기본값은 `false`이며, 이를 명시적으로 `true`로 설정하는 경로가 없다. → **R:R 필터는 사실상 항상 통과**.

**QT 분석**: R:R < 1.3인 신호가 여과 없이 전송되면 기대수익 음수 구간에서 진입이 발생한다.

**수정 방향**
1. `queue_worker.py`에서 `signal.get("tp1_price")` 등으로 R:R 직접 계산, 1.3 미만이면 CANCEL
2. 또는 Java `SignalService`에서 TP/SL 계산 후 `rr_ratio < 1.3` → `skipEntry=true` 설정

**확인 필요**
- `scorer.py`의 `S3_INST_FRGN`, `S5_PROG_FRGN` 케이스 로직
- 입력 데이터 필드(`inst_buy`, `frgn_buy`, `prog_buy` 등) 실제 수신 여부
