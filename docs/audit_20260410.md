### [C-1] S4 후보 풀 Java 미적재 — `candidates:s4:*` 항상 빈 리스트

**파일**: `api-orchestrator/.../CandidateService.java` / `ai-engine/strategy_runner.py:210-213`

**현상**

```python
# strategy_runner.py:210
kospi  = await rdb.lrange("candidates:s4:001", 0, 99)   # 항상 []
kosdaq = await rdb.lrange("candidates:s4:101", 0, 99)   # 항상 []
candidates = list(dict.fromkeys(kospi + kosdaq))[:30]   # 상위 30개 → 0개
```

`CandidateService.java`에 `getS4Candidates()` 메서드가 존재하지 않는다.  
`TradingScheduler.preloadCandidatePools()`에서도 S4 적재 호출 없음.  
Python `candidates_builder.py`가 S4 풀을 적재하는 로직(`_build_s4()`)은 있으나,  
`strategy_runner.py`와 독립 프로세스이므로 실행 타이밍 의존성이 생긴다.

**영향**: S4_BIG_CANDLE 전략이 매일 0건 스캔. 장대양봉 신호 미발행.

**수정 방향**
1. `CandidateService.java`에 `getS4Candidates()` 추가 (ka10027 or ka10032 사용)
2. `TradingScheduler.preloadCandidatePools()` 크론에 S4 적재 추가

### [H-1] 6개 전략 파일에서 `validate_kiwoom_response()` 미호출

**파일**: 아래 6개 파일

| 파일 | 상태 |
|------|------|
| `ai-engine/strategy_2_vi_pullback.py` | ❌ 미호출 |
| `ai-engine/strategy_8_golden_cross.py` | ❌ 미호출 |
| `ai-engine/strategy_9_pullback.py` | ❌ 미호출 |
| `ai-engine/strategy_13_box_breakout.py` | ❌ 미호출 |
| `ai-engine/strategy_14_oversold_bounce.py` | ❌ 미호출 |
| `ai-engine/strategy_15_momentum_align.py` | ❌ 미호출 |

**현상**

Kiwoom REST API는 서버 내부 오류(500)도 HTTP 200으로 반환한다.  
`raise_for_status()`로는 감지할 수 없으며, `http_utils.validate_kiwoom_response()`가 `return_code != "0"` 또는 `"error"` 키 존재 여부를 검증한다.  
위 6개 파일은 이 검증을 생략하고 있어, API 오류 응답을 정상 데이터로 처리할 수 있다.

호출 중인 파일(참고): `candidates_builder.py`, `strategy_1_gap_opening.py`, `strategy_3_inst_foreign.py`, `strategy_4_big_candle.py`, `strategy_10_new_high.py`, `strategy_11_frgn_cont.py`, `strategy_12_closing.py` 등.

**영향**: API 오류 시 잘못된 지표값(0, None)을 기반으로 신호가 생성되거나, KeyError로 스캔 루프 중단 가능.

**수정 방향**: 각 파일의 Kiwoom API 응답 직후 `validate_kiwoom_response(data, api_id)` 호출 추가.

### [H-2] `strategy_4_big_candle.py`, `strategy_10_new_high.py` KIWOOM_BASE_URL 기본값이 mockapi

**파일**:
- `ai-engine/strategy_4_big_candle.py:23`
- `ai-engine/strategy_10_new_high.py:23`

**현상**

```python
# strategy_4_big_candle.py:23, strategy_10_new_high.py:23
KIWOOM_BASE_URL = os.getenv("KIWOOM_BASE_URL", "https://mockapi.kiwoom.com")
```

반면 `http_utils.py:14`, `candidates_builder.py:20`는 기본값이 `https://api.kiwoom.com`.  
`config.py`는 아예 환경변수를 읽지 않고 `https://api.kiwoom.com` 하드코딩.

현재 `.env`에 `KIWOOM_BASE_URL=https://api.kiwoom.com`이 설정되어 있어 운영 중에는 문제없으나,
`.env` 미로드 환경(CI, 신규 서버 배포)에서는 S4·S10이 mock API를 호출하여 빈 데이터를 반환한다.

**영향**: 배포 환경 변경 시 S4·S10 신호 미발행. 디버그 어려움.

**수정 방향**: 두 파일의 기본값을 `"https://api.kiwoom.com"`으로 변경, 또는 `http_utils.KIWOOM_BASE_URL`을 공통 import로 통일.

### [H-3] S4 scorer 임계값과 전략 생성 임계값 불일치

**파일**: `ai-engine/strategy_4_big_candle.py:85` / `ai-engine/scorer.py:191`

**현상**

```python
# strategy_4_big_candle.py:85 – 신호 생성 조건
if not is_bull or body_ratio < 0.65 or gain_pct < 2.5:
    return None
```

```python
# scorer.py:191 – 점수 부여 기준
_body_sc = 20 if body_ratio >= 0.8 else (10 if body_ratio >= 0.7 else 0)
```

`body_ratio`가 0.65~0.69 구간에 있는 신호는 전략 파일에서 통과하지만, scorer에서 0점을 받는다.  
S4 임계값은 75점이므로, 이 구간의 신호는 체계적으로 CANCEL 처리된다.

**영향**: 0.65~0.69 body_ratio 신호가 항상 필터링됨. 불필요한 API 호출 낭비.

**수정 방향**: 둘 중 하나를 맞춤.
- 전략 파일을 `body_ratio < 0.70`으로 강화 (scorer 기준에 맞춤), 또는
- scorer 점수 기준을 `body_ratio >= 0.65` (5점) 구간 추가.

### [H-4] candidates TTL과 strategy_runner 스캔 주기 불일치 (S1, S7)

**파일**: `api-orchestrator/.../service/CandidateService.java:155` / `ai-engine/strategy_runner.py:36`

**현상**

| 전략 | candidates TTL | strategy_runner 활성 시간 |
|------|---------------|--------------------------|
| S1 (`candidates:s1:*`) | **3분** | 08:30~09:10 (40분) |
| S7 (`candidates:s7:*`) | **3분** | 08:30~09:00 (30분) |

`strategy_runner`는 60초 주기로 실행되므로, 3분 TTL 내에 Java가 재호출하지 않으면 풀이 소멸한다.  
장전 `startPreMarketSubscription(07:30)` 이후 API가 새 데이터를 응답하지 않으면 캐시 재갱신 없이 빈 리스트로 스캔.

또한 S1은 08:30~09:10(40분)이 활성 시간이므로 3분 TTL로는 최소 13회 이상 재적재가 필요한데, Java 스케줄러에서 이 시간대에 재호출 로직이 없다.

**영향**: 09:00 이후 S1 신호가 간헐적으로 미발행될 수 있음.

**수정 방향**: S1 TTL을 10분으로 연장, 또는 `strategy_runner.py`에서 S1 풀이 비어있을 때 직접 API 폴백 호출 추가.

### [M-1] `CLAUDE_API_KEY` 초기화 시점 불일치 — 런타임 CRASH 가능

**파일**: `ai-engine/engine.py:169-173` / `ai-engine/analyzer.py:37-40`

**현상**

```python
# engine.py:169 – WARNING만 출력하고 프로세스 계속 기동
if not os.getenv("CLAUDE_API_KEY"):
    logger.warning("CLAUDE_API_KEY 미설정 – 기능 제한됨.")

# analyzer.py:37 – 실제 호출 시점에 RuntimeError CRASH
api_key = os.getenv("CLAUDE_API_KEY")
if not api_key:
    raise RuntimeError("CLAUDE_API_KEY 환경 변수 미설정")
```

`engine.py`는 시작 시 WARNING만 내고 프로세스를 계속 실행한다.  
이후 신호가 들어와 `analyzer.py`가 호출되는 순간 `RuntimeError`가 발생하며, `queue_worker`의 해당 신호 처리가 중단된다.

**영향**: `CLAUDE_API_KEY` 누락 환경에서 신호를 처리하다가 예외 발생. 신호 유실 가능.

**수정 방향**: `engine.py` 기동 시점에 `CLAUDE_API_KEY` 누락이면 즉시 `sys.exit(1)` 처리하거나, analyzer 호출 경로를 분리.

### [M-2] `strategy_runner.py`와 `ForceCloseScheduler.java`에 `_SWING_STRATEGIES` 이중 관리

**파일**: `ai-engine/strategy_runner.py:43-47` / `api-orchestrator/.../ForceCloseScheduler.java:43-48`

**현상**

스윙 전략 목록이 두 곳에 별도로 정의되어 있다.

```python
# strategy_runner.py:43
_SWING_STRATEGIES = {
    "S8_GOLDEN_CROSS", "S9_PULLBACK_SWING", "S10_NEW_HIGH",
    "S11_FRGN_CONT", "S12_CLOSING", "S13_BOX_BREAKOUT",
    "S14_OVERSOLD_BOUNCE", "S15_MOMENTUM_ALIGN",
}
```

```java
// ForceCloseScheduler.java:43
private static final Set<TradingSignal.StrategyType> SWING_STRATEGIES = EnumSet.of(
    S12_CLOSING, S8_GOLDEN_CROSS, S9_PULLBACK_SWING, S15_MOMENTUM_ALIGN

```

Python에는 8개, Java에는 4개. 한 쪽을 수정해도 다른 쪽이 동기화되지 않는다. [C-2]의 근본 원인.

**영향**: 전략 추가/변경 시 두 곳 동시 수정 필요 → 유지보수 오류 반복 위험.

**수정 방향**: 스윙 전략 목록을 Redis 키(`strategy:swing_list`)에 단일 관리하거나, 공통 설정 파일로 분리.

### [M-3] `config.py`가 환경변수를 읽지 않고 하드코딩

**파일**: `ai-engine/config.py:1`

**현상**

```python
# config.py
KIWOOM_BASE_URL = "https://api.kiwoom.com"
```

`os.getenv()` 없이 하드코딩. 다른 파일들은 각자 `os.getenv("KIWOOM_BASE_URL", ...)`을 사용하므로, `config.py`를 import해도 환경변수 오버라이드가 불가하다.

**영향**: `config.py`를 사용하는 코드가 추가될 경우 환경별 URL 분리가 불가능.

**수정 방향**: `KIWOOM_BASE_URL = os.getenv("KIWOOM_BASE_URL", "https://api.kiwoom.com")`으로 변경.

### [M-4] Telegram Bot `AI_ENGINE_URL` 환경변수 기본값이 localhost

**파일**: `telegram-bot/src/services/kiwoom.js:145` (추정)

**현상**

```javascript
const AI_ENGINE_URL = process.env.AI_ENGINE_URL || 'http://localhost:8082';
```

Docker 컨테이너 환경에서 `localhost`는 컨테이너 자신을 가리키므로, `AI_ENGINE_URL` 미설정 시 ai-engine 컨테이너에 접근 불가.

**영향**: 배포 환경 변경 시 Claude 분석 호출 실패. 에러 로그 없이 타임아웃.

**수정 방향**: `docker-compose.yml`의 서비스명으로 기본값 설정 (`http://ai-engine:8082`), 또는 환경변수 없으면 기동 시 경고 출력.
