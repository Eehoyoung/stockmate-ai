# StockMate AI 본투자 가능성 확보를 위한 10라운드 멀티에이전트 토론 정리

작성일: 2026-04-25  
메인 진행 및 중재: Codex  
형식: 트레이더 3명 + 개발자 3명 관점 토론, 총 10라운드

## 1. 참가 에이전트

### 트레이더 에이전트

- `T1`: 초단타 모멘텀 실전파
- `T2`: 스윙/포지션 품질 중심 퀀트
- `T3`: 리스크 매니저 출신 포트폴리오 운용자

### 개발자 에이전트

- `D1`: 플랫폼/백엔드 신뢰성 엔지니어
- `D2`: 데이터/ML 검증 엔지니어
- `D3`: 제품/운영 자동화 엔지니어

## 2. 메인 중재자의 출발점

토론 전 메인에서 확인한 현재 상태는 아래와 같다.

- 시스템 구조는 이미 운영형에 가깝다.
  - `api-orchestrator`, `ai-engine`, `websocket-listener`, `telegram-bot`로 역할이 분리되어 있다.
  - Redis/DB/Flyway/헬스체크/포지션 모니터링/리스크 이벤트 구조가 존재한다.
- 그러나 본투자 판단을 막는 실증 데이터가 아직 약하다.
  - [docs/tp_sl_single_tp_rr_backtest_2026-04-24.md](/C:/Users/LeeHoYoung/IdeaProjects/t/stockmate-ai/docs/tp_sl_single_tp_rr_backtest_2026-04-24.md:1) 기준, 일부 최근 신호는 비용 반영 후 R:R 적합성이 약하다.
  - [docs/project_deep_dive_report_2026-04-23.md](/C:/Users/LeeHoYoung/IdeaProjects/t/stockmate-ai/docs/project_deep_dive_report_2026-04-23.md:1) 기준, 전략/운영/데이터 경계에서 복수의 구조 리스크가 이미 식별돼 있다.
- 테스트 신뢰도는 아직 투자자 설득 수준이 아니다.
  - `ai-engine` pytest는 `tests/test_scorer.py` 수집 단계에서 `SyntaxError`로 실패했다.
  - `websocket-listener` pytest는 async plugin 미설정 상태로 `3 failed, 45 passed`였다.
  - `telegram-bot`의 `tests/test_formatter.js`는 통과했다.

중재 목표는 단순 비판이 아니라, `어떤 업데이트가 완료되면 본투자 심사에서 통과 가능성이 생기는가`를 합의하는 것이다.

## 3. 10라운드 토론 요약

### 라운드 1. 각자 현재 서비스에 대한 1차 판정

- 공통 평가:
  - 아이디어 단계는 아니다.
  - 운영형 프로토타입이자 실험 플랫폼으로는 의미가 있다.
  - 하지만 지금은 본투자 대상 운용 시스템으로 보기 어렵다.
- 1차 합의:
  - 문제는 기능 부족보다 `투자자 신뢰 부족`이다.
  - 핵심 결핍은 `실체결 검증`, `기대값 검증`, `리스크 통제`, `운영 신뢰성`이다.

### 라운드 2. 관점 충돌 정리

- `T1`은 execution이 신호보다 먼저라고 주장했다.
- `T2`는 기대값과 워크포워드 검증이 먼저라고 주장했다.
- `T3`는 포트폴리오 레벨 리스크 엔진과 킬스위치를 먼저 요구했다.
- `D1`은 idempotency, 큐 무결성, 장애 복원력을 우선순위로 올렸다.
- `D2`는 데이터 계약, 라벨 고정, 재현성 없는 성과는 무의미하다고 봤다.
- `D3`는 운영 관제, 승인 흐름, runbook 부재를 강하게 지적했다.
- 2차 합의:
  - `좋은 전략` 하나만으로는 부족하다.
  - `실행 후에도 남는 기대값`과 `실패 시 자동 중단`이 동시에 있어야 한다.

### 라운드 3. 전략 수 축소 논쟁

- `T1`: 15개 전략은 너무 많고, 3개 이하로 줄여야 한다.
- `T2`: 15개 유지보다 5~7개 전략만 남기고 OOS 기준으로 재평가해야 한다.
- `T3`: 전략 이름이 달라도 상관관계와 익스포저가 겹치면 하나로 봐야 한다.
- 합의:
  - 전략 수는 줄인다.
  - 우선 전략군은 `고유동성`, `체결 용이`, `비용 반영 후 EV 양수`, `레짐 강건성` 기준으로 선별한다.
  - `저유동성 급등 추격형`, `상관 높은 유사 전략`, `운영 복잡도만 높이는 전략`은 후순위로 밀린다.

### 라운드 4. 검증 방법 논쟁

- `T1`: 운영 환경과 동일 경로로 shadow trading을 돌려야 한다.
- `T2`: walk-forward와 out-of-time 검증이 먼저다.
- `D1`: 순서는 `전략이 되는지` 확인 후 `운영해도 안 망가지는지` 확인이 맞다고 정리했다.
- `D2`: 비용 반영 라벨과 평가 체계를 먼저 고정해야 한다고 강조했다.
- 합의:
  - 순서는 `라벨/데이터/비용 모델 고정 -> walk-forward 검증 -> shadow trading -> 소액 실거래`다.

### 라운드 5. 투자자에게 보여줄 핵심 증거 논쟁

- `T1`: 실제 체결 슬리피지 분포, 손절 집행 성공률, 30거래일 이상 execution-adjusted PnL 일관성이 핵심이라고 봤다.
- `T2`: 비용 반영 OOS EV, 워크포워드 누적 성과, 레짐 강건성이 핵심이라고 봤다.
- `T3`: 한도 위반 0회, 체결 반영 기대값, 자동 안전정지 검증을 요구했다.
- `D1/D2/D3`: 유실 0건, 중복 0건, 추적 가능성, 운영 SLO, 재현성 증거를 요구했다.
- 합의:
  - 투자자는 설명보다 `증거 패키지`를 원한다.
  - 그 패키지는 `수익성`, `실행성`, `안전성`, `재현성`, `운영성`으로 나뉘어야 한다.

### 라운드 6. 30/60/90일 로드맵 압축

- `30일`:
  - 전략 축소
  - 데이터/라벨/비용 모델 고정
  - 깨진 테스트 복구
  - event id 및 trace 체계 도입
  - 중앙 리스크 엔진 최소 기능 도입
  - 운영 대시보드 최소판 구축
- `60일`:
  - rolling walk-forward
  - 계약 테스트 및 장애 주입 테스트
  - shadow trading 20거래일 이상 운영
  - 섹터/상관관계 한도 추가
  - human-in-the-loop, retry console, alert center 추가
- `90일`:
  - 소액 실거래 검증
  - 실체결과 백테스트 괴리 분석
  - 리스크 위반 0회 운영 이력 축적
  - 투자자용 검증 패키지 확정

### 라운드 7. 투자자가 던질 가장 아픈 질문 정리

- `실제 체결 기준으로도 돈이 남는가?`
- `과최적화가 아니라는 증거가 있는가?`
- `시스템 오류와 전략 실패를 어떻게 분리해 설명하는가?`
- `장애 시 누가 어떻게 막고 복구하는가?`
- `시장 국면이 바뀌면 언제 어떻게 멈추는가?`

합의된 답변 방향은 한 문장으로 정리된다.

`백테스트 설명이 아니라, 비용 반영 OOS 검증 + shadow live 로그 + 리스크/운영 통제 로그로 답해야 한다.`

### 라운드 8. 본투자 직전 최종 승인 체크포인트

- execution:
  - 30거래일 이상 execution-adjusted 성과
  - 손절 집행 성공률 기준 충족
  - 실시간 stale data 차단
- quant:
  - OOS EV 플러스
  - walk-forward 강건성
  - shadow live와 백테스트 괴리 허용 범위 내
- risk:
  - 종목/전략/계정/섹터 한도 강제
  - 킬스위치 실동작 검증
- reliability:
  - 유실 0건
  - 중복 0건
  - 상태 불일치 0건
- ops:
  - 운영 SLO 충족
  - human-in-the-loop 실제 사용 로그
  - 장애 drill 통과

### 라운드 9. 절대 양보 불가 조건 정리

6개 에이전트의 공통 최소조건은 다음 5개 축으로 수렴했다.

- `실체결 반영 검증`
- `비용 반영 OOS 기대값 증명`
- `중앙 리스크 엔진 + 킬스위치`
- `idempotency + 큐 무결성 + 장애 복구`
- `운영 관제 + 수동 승인 + 재처리 통제`

### 라운드 10. 최종 투자 입장

- 6명 모두 현재 시점 최종 입장은 `유보`였다.
- 다만 의미 있는 변화가 있었다.
  - 시작 시점의 유보는 `지금은 투자 불가에 가까운 유보`였다.
  - 10라운드 종료 시점의 유보는 `핵심 로드맵 완료 후 단계적 본투자 검토가 가능한 유보`로 이동했다.

즉, 토론의 결론은 다음과 같다.

`지금 바로 본투자는 어렵다. 그러나 아래 업데이트가 실제 데이터와 운영 로그로 입증되면, 본투자 검토가 가능한 상태까지는 충분히 올릴 수 있다.`

## 4. 최종 합의 업데이트 항목

### A. 전략 및 포트폴리오

1. 전략 수를 `15개 -> 3~7개` 수준으로 축소한다.
2. 생존 기준을 `고유동성`, `체결 가능성`, `비용 반영 EV`, `레짐 강건성`, `포트폴리오 기여도`로 통일한다.
3. 상관 높은 유사 전략은 통합하거나 제거한다.
4. 전략 단위 평가가 아니라 포트폴리오 단위 MDD와 상관구조로 최종 선별한다.

### B. 체결 및 실전 검증

1. `signal -> order_candidate -> accepted -> filled -> exited` 전 구간 타임스탬프 로깅을 강제한다.
2. 틱/호가 기반 execution simulator를 만든다.
3. shadow trading은 운영과 동일한 큐/리스크 경로를 통과하게 만든다.
4. 전략별 `슬리피지`, `fill rate`, `부분체결`, `미체결`, `손절 성공률`, `초과손실`을 일일 리포트한다.
5. 본투자 전 최소 `20~30거래일` 이상의 shadow 또는 소액 실거래 이력을 확보한다.

### C. 퀀트/모델 검증

1. 라벨 정의를 고정한다.
  - 진입 시점
  - 청산 규칙
  - 보유기간
  - TP/SL
  - 비용 반영 규칙
2. 비용 반영 백테스트 규칙을 단일화한다.
3. `rolling walk-forward`, `out-of-time`, `레짐별 성과 분해`, `민감도 테스트`를 기본 검증 세트로 고정한다.
4. 투자자용 핵심 지표를 표준화한다.
  - `EV`
  - `Sharpe`
  - `Sortino`
  - `MDD`
  - `profit factor`
  - `precision at top-k`
  - `calibration`
  - `cost after fee/slippage`
5. 데이터 버전, feature/label 버전, 코드 SHA, seed, artifact를 남기는 재현성 인프라를 붙인다.

### D. 중앙 리스크 엔진

1. 주문 전 한도 검증을 중앙화한다.
  - 종목별
  - 전략별
  - 계정별
  - 섹터별
2. 일중 손실 킬스위치를 구현한다.
3. 월간 MDD 기준 감속 또는 중단 규칙을 구현한다.
4. 시세 stale, heartbeat 누락, 주문 응답 불일치, Redis/DB 장애 시 신규 진입을 자동 차단한다.
5. 전략별 성과 악화 시 주문 크기 축소 또는 비활성화 규칙을 추가한다.

### E. 신뢰성 및 배포

1. 모든 서비스 테스트를 green 상태로 복구한다.
2. 서비스 간 payload 계약 테스트를 추가한다.
3. Redis/WebSocket/DB 장애 주입 테스트를 추가한다.
4. 전 구간 idempotency 키를 도입한다.
5. DLQ, retry, timeout, replay 정책을 코드와 문서에 고정한다.
6. canary, rollback, migration gate를 포함한 배포 게이트를 강제한다.

### F. 운영 제품화

1. 단일 운영 관제판을 만든다.
2. Signal pipeline trace 화면을 만든다.
3. Alert center와 incident 화면을 만든다.
4. Queue/retry console을 만든다.
5. human-in-the-loop를 실제 운영 흐름으로 연결한다.
  - 고위험 신호 승인
  - 자동 중단 후 수동 재개
  - 중복/충돌 신호 검수
  - backlog 재처리 승인
6. runbook과 출시 체크리스트를 화면 및 로그와 연결한다.

## 5. 투자자가 본투자를 검토할 수 있는 승인 조건

- 종목/전략/계정/섹터 한도 강제

### 필수 조건 4. 신뢰성

- 유실 0건
- 중복 0건
- 상태 불일치 0건
- 장애 복구 리허설 통과

### 필수 조건 5. 운영성

- 운영 대시보드 상시 가동

## 7. 메인 중재자 최종 정리

이 서비스는 `버릴 프로젝트`가 아니라 `심사 가능한 수준으로 끌어올릴 프로젝트`에 가깝다.  
다만 본투자 가능성은 기능 추가가 아니라 아래 전환에서 생긴다.

- `전략 수 확대 -> 검증된 전략 집중`
- `백테스트 수익률 -> 체결 반영 기대값`
- `개별 전략 -> 포트폴리오 통제`
- `작동하는 시스템 -> 망가지면 멈추는 시스템`
- `문서 설명 -> 운영 로그와 실증 패키지`

현재 결론은 `즉시 본투자 반대에 가까운 유보`다.  
목표 상태는 `핵심 업데이트 완료 후 단계적 본투자 검토가 가능한 유보`이며, 이 문서의 업데이트 항목은 그 상태로 이동시키기 위한 합의안이다.
