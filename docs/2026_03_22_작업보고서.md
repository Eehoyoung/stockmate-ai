**작업일시**: 2026-03-22

## 작업 개요

전략 스캐너(`strategy_runner.py`)에 Telegram 직접 알림 기능을 추가하고, `api-orchestrator`의 신호 DTO에 2차 목표가 및 실제 가격 계산 기능을 구현하였다.

---

## 분석 결과

### strategy_runner.py 현황

| 항목 | 내용 |
|------|------|
| 위치 | `ai-engine/strategy_runner.py` |
| 역할 | Java api-orchestrator의 보완 경로로 S1~S7 전략을 Python에서 직접 실행하여 `telegram_queue`(Redis)에 신호를 발행 |
| 실행 방식 | `ENABLE_STRATEGY_SCANNER=true` 설정 시 `engine.py`가 별도 asyncio 태스크로 실행; 60초 간격 폴링 |
| 병렬 제어 | `asyncio.Semaphore(MAX_CONCURRENT_STRATEGIES)`로 동시 실행 수 제한 |
| **문제점 1** | Redis 큐 발행 후 Telegram 직접 알림 로직 없음 — 신호 발생 시점에 즉시 통보 불가 |
| **문제점 2** | `TelegramNotifier` 클래스 미존재 |
| **문제점 3** | 전략 신호 dict에 2차 목표가(`target2_pct`) 필드 없음 |

## 수행한 작업

### ai-engine/telegram_notifier.py (신규 파일)

- `TelegramNotifier` 클래스 구현
  - `TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID` (또는 `TELEGRAM_ALLOWED_CHAT_IDS`) 환경변수 참조
  - `TELEGRAM_DRY_RUN=true` 설정 시 실제 HTTP 호출 없이 로그만 출력 (테스트·모의 모드)
  - 콤마 구분 다중 chat_id 지원
  - `aiohttp` 비동기 HTTP 클라이언트로 10초 타임아웃 적용
  - 전송 실패 시 예외를 삼키고 `False` 반환 — 전략 루프 중단 방지
- `format_buy_signal(sig: dict) → str` 함수 구현
  - 입력: 전략 신호 dict (`stk_cd`, `stk_nm`, `strategy`, `cur_prc`/`entry_price`, `target_pct`, `target2_pct`, `stop_pct` 등)
  - Telegram HTML 형식 매수 추천 알림 메시지 생성
  - `target2_pct` 미설정 시 `target_pct * 1.5` 자동 계산
  - `cur_prc`(진입가) 있을 경우 실제 원화 가격 계산 (`round(base * (1 + pct/100))`)

### api-orchestrator 수정 내용

**TradingSignalDto.java:**
- `target2Pct` 필드 추가 (2차 목표 비율)
- `resolvedTarget2Pct()`: 명시 미설정 시 `targetPct * 1.5` 자동 반환
- `calcTarget1Price()`, `calcTarget2Price()`, `calcStopPrice()`: 진입가 기반 실제 가격 계산 메서드
- `toTelegramMessage()`: 원화 가격 표시 형식으로 전면 개선 (진입가·1차/2차 목표가·손절가 모두 원화+퍼센트)
- `toQueuePayload()`: `target2_pct` 필드 추가 (ai-engine에서 활용 가능)

**StrategyService.java:**
- S1 (갭상승 시초가): `entryPrice=expPrice`, `target2Pct=6.0` 추가
- S2 (VI 눌림목): `entryPrice=curPrice`, `target2Pct=4.5` 추가
- S3 (외인+기관): `target2Pct=5.0` 추가
- S4 (장대양봉): `entryPrice=c` (5분봉 종가), `target2Pct=6.0` 추가
- S5 (프로그램+외인): `target2Pct=4.5` 추가
- S6 (테마 후발주): `target2Pct=round(target * 1.5)` 추가
- S7 (동시호가): `entryPrice=expPrice`, `target2Pct=min(t1*1.5, 8.0)` 추가

**SignalService.java:**
- `buildSignalEntity()`: `targetPrice = dto.calcTarget1Price()` 및 `stopPrice = dto.calcStopPrice()` 설정 추가 (기존에 entity 컬럼은 있었으나 값이 채워지지 않았음)

## 미완료 항목 / 추천 후속 작업

1. **Java api-orchestrator Telegram 직접 발송 추가 (선택)**: 현재 Java 측은 Redis 큐(`telegram_queue`)를 통해 telegram-bot(Node.js)으로 신호를 전달한다. 요구사항에 따라 Java에서도 직접 Telegram 발송이 필요하다면 `TelegramNotifyService` 빈을 추가하고 `SignalService.processSignal()` 내에서 호출하는 구조가 적합하다.

2. **S2 VI 눌림목 target2Pct 검증**: S2는 실시간 이벤트 기반이므로 시뮬레이션 테스트가 어렵다. VI 발동 테스트 환경 구성 후 신호 포맷 검증 권장.

3. **전략 신호 dict 표준화**: Python 전략 파일(strategy_1~7.py)이 반환하는 신호 dict의 `cur_prc` / `entry_price` 필드 유무가 전략마다 다를 수 있다. 전략 파일별 반환 스키마를 문서화하거나, 공통 `SignalSchema` dataclass를 도입하여 필드 누락 위험을 제거하는 것이 권장된다.

4. **유닛 테스트 보강**: `tests/test_strategy_runner.py`(기존 미구현)에 `TelegramNotifier` dry-run 모드를 활용한 알림 포맷 검증 테스트 추가 권장.

5. **api-orchestrator 2차 목표가 DB 컬럼 추가**: `TradingSignal` 엔티티에 `target2Price` / `target2Pct` 컬럼이 없어 현재 DB에 저장되지 않는다. 스윙 전략(S8~S13) 확장 시 추가 마이그레이션 필요.

6. **JPA DDL 설정**: `application.yml`의 `ddl-auto: create`는 서버 재시작 시 테이블을 초기화한다. 운영 환경 전환 전 `update` 또는 `validate`로 변경 필요.
