# StockMate AI 운영 실행계획서

작성일: 2026-04-18  
작성 관점: 시니어 운영 엔지니어  
반영 페르소나:
- PO: 우선순위, 배포 리스크, 운영 표준화 기준
- PM: 단계별 범위 통제, 검증 완료 기준
- 퀀트 트레이더: 장전/정규장/NXT 시간창 요구사항, 신호 파이프라인 가시성 요구

## 문서 범위

이 문서는 `P0/P1/P2` 중 운영/배포/모니터링/설정 검증 관점 작업만 별도로 정리한다.  
기능 로직 고도화, 전략 규칙 변경, 메시지 UX 변경은 제외한다.

## 운영 원칙

- 모든 시간 기준은 `KST(Asia/Seoul)`로 고정한다.
- WebSocket 운영 시간은 `월-금 07:30~20:10 KST`를 표준으로 한다.
- 배포는 "기동됨"이 아니라 "필수 설정 검증 + 핵심 헬스체크 성공"까지 완료되어야 성공으로 본다.
- 운영 메시지는 신호 메시지와 분리해서 관리한다.

## P0

### 목표

- 배포 직후 죽는 서비스와 설정 누락 장애를 제거한다.
- 하루 단위로 신호 파이프라인이 어디서 막히는지 운영자가 바로 볼 수 있게 만든다.
- 서비스별 시간대와 연결 설정이 모두 KST 기준으로 고정되었는지 검증 가능하게 만든다.

### 구체 작업

1. 배포 설정 검증 게이트 도입
   - `telegram-bot`, `api-orchestrator`, `ai-engine`, `websocket-listener`에 공통 기동 전 검증 루틴을 둔다.
   - 필수 항목:
     - `KST/Asia-Seoul` 시간대 설정 여부
     - Redis/Postgres 연결값 존재 여부
     - Telegram 필수 env 존재 여부
     - API base URL 존재 여부
     - WebSocket 운영 시간 설정값 존재 여부
   - 검증 실패 시 서비스는 즉시 종료하고, 누락된 변수명을 명확히 출력한다.

2. 운영 헬스체크 표준화
   - 서비스별 `/health` 또는 동등한 내부 체크를 표준화한다.
   - 최소 체크 항목:
     - Redis 연결 성공
     - Postgres 연결 성공
     - 최근 heartbeat 존재 여부
     - 최근 queue publish/consume 가능 여부
   - Telegram 운영 명령 `/status`, `/signals`, `/health`, `/today`에서 같은 기준을 재사용한다.

3. 일일 파이프라인 집계 추가
   - 전략별로 다음 수치를 일 단위 집계한다.
     - 후보 생성수
     - 규칙 통과수
     - AI 통과수
     - Telegram 발송수
     - timeout 수
     - cancel 사유 비중
   - 로그만 뒤지지 않도록 일일 요약을 파일 또는 DB 테이블로 남긴다.

4. 배포 후 스모크 체크리스트 고정
   - 재배포 직후 5분 안에 확인할 항목을 문서화하고 스크립트화한다.
   - 포함 항목:
     - 서비스 프로세스 기동 여부
     - WebSocket 연결 상태
     - Telegram 명령 응답 여부
     - Redis key write/read 여부
     - 오늘 날짜 기준 KST 로그 타임스탬프 정상 여부

### 검증 방법

- `.env` 또는 배포 secret에서 필수 값 하나를 제거했을 때 서비스가 의도대로 즉시 실패해야 한다.
- 운영자가 Telegram `/status`, `/signals`, `/health`를 호출했을 때 5초 안에 응답을 받아야 한다.
- 배포 후 `KST` 기준 시간이 로그에 일관되게 기록되어야 한다.
- 하루치 집계에서 전략별 `candidate -> rule -> AI -> send` 숫자가 누락 없이 채워져야 한다.

## P1

### 목표

- 장전/장중 운영 품질을 실시간으로 판단할 수 있게 만든다.
- timeout, warm-up 미완료, WS 세션 이탈 같은 운영성 이슈를 "발생 후 추적"이 아니라 "발생 중 감지" 상태로 전환한다.
- 표준 운영 시간 정책을 코드, 설정, 문서에 일치시킨다.

### 구체 작업

1. WebSocket 세션 정책 문서화 및 강제
   - 표준 운영 시간: `월-금 07:30~20:10 KST`
   - 장전 동시호가, 정규장, NXT 대응 목적을 설정 파일과 문서에 명시한다.
   - 이 시간 외 연결은 예외 플래그 없이는 열리지 않도록 한다.
   - 시작/종료 시각, 재연결 횟수, 최근 heartbeat age를 모니터링 지표로 남긴다.

2. 장전 warm-up 체크 도입
   - `07:20~08:20 KST` 사이에 자동 점검을 수행한다.
   - 점검 항목:
     - Redis 접근 가능
     - 후보풀 사전 적재 완료 여부
     - 종목명 캐시 접근 가능
     - WebSocket heartbeat 정상 여부
   - warm-up 실패 시 운영 경고를 보내고, 실패 항목을 분리 기록한다.

3. timeout 원인 분해 로그 표준화
   - timeout을 단순 횟수로만 보지 않고 다음 범주로 나눈다.
     - 데이터 대기
     - 외부 API 대기
     - 연산 병목
     - 큐 적체
   - 전략별 timeout override 사용 여부와 실제 경과시간을 같이 남긴다.

4. STATUS_REPORT 정책 고정
   - 운영 브리핑은 `장전 준비`, `점심 점검`, `장마감 요약` 3개 목적만 허용한다.
   - 기본 슬롯은 `08:30`, `12:00`, `15:40 KST`로 유지한다.
   - 신호 채널과 운영 채널이 섞이지 않도록 발송 대상을 분리한다.

5. 운영 대시보드 지표 정리
   - 최소 지표:
     - WS heartbeat age
     - WS reconnect count
     - queue backlog
     - 전략별 timeout 수
     - 전략별 candidate/rule/AI/send 수
     - Redis/Postgres 연결 상태
   - 운영자는 콘솔 접속 없이 Telegram 명령 또는 로그 요약만으로 이상 여부를 파악할 수 있어야 한다.

### 검증 방법

- 장외 시간에는 WS가 연결되지 않고, `07:30 KST` 이후에는 자동 연결되어야 한다.
- `07:20~08:20 KST` 사이 warm-up 결과가 기록되어야 하고 실패 시 경고가 발송되어야 한다.
- timeout 로그에서 전략명, 소요시간, 원인 범주를 바로 확인할 수 있어야 한다.
- `STATUS_REPORT`가 지정 슬롯 외 시간에는 발송되지 않아야 한다.
- 대시보드 또는 일일 요약에서 WS/queue/timeout/candidate 지표가 모두 집계되어야 한다.

## P2

### 목표

- 운영 회귀를 줄이고, 반복 가능한 배포/리뷰 체계를 만든다.
- 장애 대응을 사람 기억이 아니라 체크리스트와 정기 루프로 관리한다.
- 설정 변경, 시간창 변경, 배포 정책 변경이 다른 서비스에 미치는 영향을 사전에 확인한다.

### 구체 작업

1. 변경관리 체크리스트 도입
   - PR 또는 배포 문서에 다음 체크리스트를 강제한다.
     - env 변경 여부
     - 시간대 변경 여부
     - 스케줄 변경 여부
     - Redis key 정책 변경 여부
     - Telegram 명령 영향 여부
     - WS 운영 시간 영향 여부

2. 주간 운영 회고 루프 고정
   - 매주 다음 항목을 30분 내 검토한다.
     - 발송 수 변화
     - timeout 상위 전략
     - warm-up 실패 사례
     - WS 재연결 급증 사례
     - 설정 누락/배포 실패 사례
   - 결과는 다음 주 작업 우선순위에 반영한다.

3. 운영 런북 정리
   - 장애 유형별 즉시 대응 절차를 문서화한다.
   - 최소 런북:
     - Telegram 명령 실패
     - WS heartbeat stale
     - Redis 연결 실패
     - Postgres 연결 실패
     - 장전 warm-up 실패
     - 전략 timeout 급증

4. 배포 자동화 고도화
   - 장기적으로는 수동 실행이 아니라 배포 스크립트에서 다음 순서를 강제한다.
     - secret 주입 확인
     - 설정 검증
     - 서비스 기동
     - 스모크 테스트
     - 실패 시 즉시 롤백 또는 중단

### 검증 방법

- 변경사항이 있는 PR마다 운영 체크리스트가 채워져 있어야 한다.
- 주간 회고 문서에서 전주 대비 장애/timeout/재기동 추세를 확인할 수 있어야 한다.
- 런북만 보고도 운영자가 1차 대응을 수행할 수 있어야 한다.
- 배포 자동화 단계에서 설정 누락이 있으면 본 실행 전에 차단되어야 한다.

## 우선순위 요약

- `P0`: 배포 설정 검증, 운영 헬스체크, 일일 파이프라인 집계
- `P1`: WS 시간 정책 강제, 장전 warm-up, timeout 원인 분해, STATUS_REPORT 고정
- `P2`: 변경관리 체크리스트, 주간 회고, 운영 런북, 배포 자동화

## 완료 기준

- 운영자가 Telegram과 로그 요약만으로 서비스 상태를 5분 안에 파악할 수 있다.
- 설정 누락은 기동 전에 차단된다.
- 장전/장중/NXT 시간 정책이 코드, 설정, 문서에서 모두 일치한다.
- timeout, WS 이탈, warm-up 실패가 "사후 분석"이 아니라 "실시간 운영 지표"로 보인다.
