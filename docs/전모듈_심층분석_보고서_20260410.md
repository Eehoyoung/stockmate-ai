### [C-1] S4_BIG_CANDLE 후보 풀 미적재 — 신호 영구 0건

**관련 파일**
- `ai-engine/strategy_runner.py:207–224`
- `api-orchestrator/.../service/CandidateService.java`
- `api-orchestrator/.../scheduler/TradingScheduler.java`

**현상**

`strategy_runner.py`에서 S4 스캔 시 `candidates:s4:001` / `candidates:s4:101` 키를 읽는다.


그러나 `CandidateService.java`에 `getS4Candidates()` 메서드가 **존재하지 않는다.**  
`TradingScheduler.preloadCandidatePools()`에서도 S4를 적재하지 않는다.  
→ `candidates:s4:*` 키는 항상 빈 리스트 → **S4 장대양봉 신호 영구 미발생.**

**페르소나별 분석**

- **SE:** CandidateService에 S8~S15 전략 전용 메서드는 모두 구현되어 있지만 S4만 누락된 상태. `TradingScheduler.preloadCandidatePools()`의 S4 적재 라인도 없다.
- **QT:** S4는 거래량 급증 + 장대양봉이라는 가장 강력한 단기 모멘텀 신호 중 하나. 이 신호가 완전히 죽어있으면 고품질 추격 진입 기회를 100% 놓친다.

**수정 방향**

1. `CandidateService.java`에 `getS4Candidates()` 추가 — `ka10023`(거래량급증) 또는 `ka10030`(당일거래량상위) 기반, `fluRt >= 3.0 && fluRt <= 20.0`, `limit(100)`
2. `TradingScheduler.preloadCandidatePools()` 내 S4 풀 적재 라인 추가
3. Redis 키: `candidates:s4:{market}`, TTL: 5분 (단기 전략 기준)

### [C-3] `hibernate.ddl-auto: update` + Flyway 병행 — 스키마 일관성 파괴

**관련 파일**
- `api-orchestrator/src/main/resources/application.yml:8,11–16`

**현상**

```yaml
# application.yml
jpa:
  hibernate.ddl-auto: update      # Hibernate가 엔티티 기반으로 DDL 자동 수정
flyway:
  enabled: true                   # Flyway도 migration으로 DDL 관리
  validate-on-migrate: false      # 충돌 검증 비활성화
```

**두 개의 DDL 엔진이 동시에 스키마를 제어**하는 상태다.

**페르소나별 분석**

- **DB:**
  - Hibernate `update`는 JPA 엔티티에 새 필드 추가 시 컬럼을 자동 생성하지만, Flyway `flyway_schema_history`에 기록을 남기지 않는다.
  - 결과: 로컬 dev DB와 staging/prod DB 스키마가 점점 diverge. 재현 불가능한 환경 차이 발생.
  - `validate-on-migrate: false` 설정이 이 충돌을 감지조차 못하게 한다.
  - 장기적으로 `flying_schema_history`가 실제 DB 상태를 반영하지 못하게 되어 Flyway 마이그레이션 실패 위험.

**수정 방향**

Flyway를 DDL의 단일 소유자로 만든다.

```yaml
# application.yml 수정
jpa:
  hibernate.ddl-auto: validate   # Flyway가 DDL 소유, Hibernate는 검증만

flyway:
  enabled: true
  validate-on-migrate: true      # 스키마 일치 검증 활성화 (불일치 시 앱 기동 차단)
  baseline-on-migrate: true
  baseline-version: 1
```

신규 컬럼·테이블 추가 시 반드시 `db/migration/V{N}__{설명}.sql` 작성.

### [H-1] `skip_entry` R:R 필터 미연결 — 실질적 무기능

**관련 파일**
- `ai-engine/queue_worker.py:131–144`

**현상**

`queue_worker.py`에서 R:R 미달 취소 로직:

```python
# queue_worker.py:131-144
if signal.get("skip_entry"):
    # TP/SL 계산 시 R:R < MIN_RR_RATIO → 진입 취소
    rr_val     = signal.get("rr_ratio", 0.0)
    action     = "CANCEL"
    ...
else:
    action     = "ENTER"   # ← skip_entry가 False면 항상 여기
```

`skip_entry=True`는 `tp_sl_engine.py`가 TP/SL 계산 후 R:R < 1.3이면 설정해야 한다. 그러나 `queue_worker.py` 내에 `tp_sl_engine` 호출이 없다.

Java `TradingSignal.skipEntry` 기본값은 `false`다 (`@Builder.Default private Boolean skipEntry = false`). Java가 이를 명시적으로 True로 설정하는 경로가 없으면 → **R:R 필터는 사실상 항상 통과.**

**수정 방향**

두 가지 중 하나:
1. `queue_worker.py`에서 `signal.get("tp1_price")` 등을 이용해 R:R을 직접 계산하고 1.3 미만이면 CANCEL 처리
2. Java `SignalService`에서 TP/SL 계산 후 `rr_ratio < 1.3`이면 `skipEntry=true`를 설정하여 payload에 포함

### [H-2] S14_OVERSOLD_BOUNCE ATR 가산점 역방향 — 리스크 관리 위반

**관련 파일**
- `ai-engine/scorer.py:365`

**현상**


과매도 반등 전략에서 ATR이 높을수록 점수가 높다.

**QT 분석**

- ATR이 높다 = 변동성이 크다 = SL을 더 멀리 설정해야 한다 = R:R 열화
- ATR > 3%는 패닉 구간으로 저점 확인 전 추가 하락 위험이 더 크다
- 트레이더 원칙: ATR은 TP/SL 거리 계산의 입력값이지, 진입 품질 보너스가 아니다

**수정 방향**

```python

_atr_sc = 15 if 1.0 <= atr_pct <= 2.5 else (5 if 0.5 <= atr_pct < 1.0 else (-5 if atr_pct > 3.0 else 0))
```

### [L-3] OpenPosition `signal_id UNIQUE` 제약 — 스케일인 진입 불가

**관련 파일**
- `api-orchestrator/.../domain/OpenPosition.java:39–40`

```java
@JoinColumn(name = "signal_id", nullable = false, unique = true)
private TradingSignal signal;
```

1:1 매핑이므로 동일 신호에 대한 분할 진입(스케일인), 물타기, 피라미딩 기록이 불가능하다.  
현재는 단일 진입 가정이므로 즉각적 문제는 없지만, 포지션 관리 고도화 시 설계 제약이 된다.  
설계 의도를 주석으로 명시적으로 남길 것.

### [L-4] TradingScheduler PRELOAD_POOL 적재 지연 누적 위험

**관련 파일**
- `api-orchestrator/.../scheduler/TradingScheduler.java:287–300`

```java
PRELOAD_POOL.submit(() -> {
    for (String mkt : new String[]{"001", "101"}) {
        getS8Candidates(mkt);  getS9Candidates(mkt);
        getS10Candidates(mkt); getS11Candidates(mkt);
        getS12Candidates(mkt); getS13Candidates(mkt);
        getS14Candidates(mkt); getS15Candidates(mkt);
    }
});
```

이 `submit`은 15분마다 실행되지만 내부 작업이 15분 이상 걸릴 경우(Kiwoom API Rate Limit, 장 초반 트래픽 집중) `PRELOAD_POOL` 큐에 미완료 작업이 누적된다. 완료 여부 확인 로직이 없다.

**수정 방향:** `Future<?>`로 결과를 저장하고 다음 실행 전 이전 작업의 완료 또는 타임아웃을 체크.
