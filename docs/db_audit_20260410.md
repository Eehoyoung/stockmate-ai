## CRITICAL

### C-1. `open_positions` 완전 비어있음 — 포지션 생성 로직 미동작

**증거**
```
trading_signals WHERE action='ENTER' → 47건
  S6_THEME_LAGGARD  ENTER EXPIRED  15건
  S11_FRGN_CONT     ENTER EXPIRED   5건
  S12_CLOSING       ENTER EXPIRED  11건
  S15_MOMENTUM_ALIGN ENTER EXPIRED 15건
  S10_NEW_HIGH      ENTER LOSS      1건

open_positions → 0건
```

**분석**  
ENTER 신호가 47건 발행됐음에도 `open_positions`에 레코드가 한 건도 없다. `signal_status`가 `EXPIRED`로 끝나는 것으로 보아 텔레그램 봇이 신호를 수신했으나, 실제 체결 후 포지션을 DB에 기록하는 로직이 실행되지 않은 것으로 추정된다.

**영향**
- `daily_pnl`의 `gross_pnl_abs`, `net_pnl_abs`, `cumulative_pnl_abs`가 모두 NULL
- `overnight_evaluations` 0건 (포지션 없으니 당연)
- `ForceCloseScheduler`, `SignalPerformanceScheduler`가 포지션 기반으로 동작하므로 실질적으로 무력화
- win_rate 집계 불가 (strategy_daily_stats의 win_rate도 NULL 또는 0)

**확인 필요**
- ForceCloseScheduler가 open_positions INSERT를 언제 수행하는지
- 텔레그램 봇에서 "실행됨" 확인 → api-orchestrator 콜백 경로 존재 여부
- 수동 주문 시스템인 경우 체결 확인 후 DB 기록하는 엔드포인트가 있는지

### C-2. `stk_cd`에 `_AL` suffix 이상

**증거**
```sql
SELECT stk_cd FROM trading_signals ORDER BY created_at DESC LIMIT 5;
-- 061970_AL
-- 028670_AL
-- 001440_AL
-- 103590_AL
-- 451760_AL
```

**분석**  
모든 종목코드에 `_AL`이 붙어있다. 키움 API의 표준 종목코드는 6자리 숫자 (`061970`)이며 `_AL` suffix는 정의된 바 없다.

**영향**  
- 키움 API를 통한 추가 조회 시 (`ka10001`, `ka10081` 등) 코드 불일치로 오류 발생 가능
- `candidate_pool_history`와 `trading_signals` 조인 시 stk_cd 불일치 가능성
- `stock_master` 적재 후 FK 참조 불가

**확인 필요**  
- `api-orchestrator`의 신호 저장 로직에서 stk_cd를 어떻게 구성하는지
- `candidates:s{N}:{market}` Redis 키에 담긴 stk_cd 포맷 확인
- `_AL` suffix가 알고리즘 거래 구분 코드인지 아니면 파싱 버그인지

## HIGH

### H-1. `ai_score == rule_score == signal_score` — Claude AI 분석 미적용

**증거**
```
strategy       | avg_ai | avg_rule | avg_sig
S11_FRGN_CONT  |   76.8 |     76.8 |    76.8
S12_CLOSING    |   66.1 |     66.1 |    66.1
S15_MOMENTUM   |   70.6 |     70.6 |    70.6
```
전 전략에서 세 점수가 소수점까지 완전히 동일.

**분석**  
`confirm_worker.py`(Claude 2차 분석)가 실제로 실행되지 않거나, Claude API 응답을 받은 후 `ai_score`를 별도로 갱신하지 않고 `rule_score`를 그대로 복사하여 저장하고 있는 것으로 추정된다.

**영향**
- Claude AI 사용 비용이 발생하는 경우 실질적인 효과 없이 비용 낭비
- CLAUDE_THRESHOLDS 기반 필터링이 rule_score 기반으로만 동작하므로 AI 판단이 신호 품질에 미반영
- `signal_score_components.strategy_components`(JSONB)에 AI 판단 근거가 누락

**확인 필요**
- `confirm_worker.py`가 `engine.py`에서 실제로 import 및 호출되는지
- `queue_worker.py` → `analyzer.py` → `confirm_worker.py` 파이프라인에서 ai_score 갱신 위치
- `CLAUDE_API_KEY` 환경변수 설정 여부

### H-2. S3, S5 점수 붕괴 — 필터링 실패 수준

**증거**
```
S3_INST_FRGN  | avg_ai: 27.3 | 53건 전부 CANCEL (passed: 0/53)
S5_PROG_FRGN  | avg_ai:  5.3 | 40건 전부 CANCEL (passed: 0/40)
```

**분석**  
S3/S5는 각각 기관/외인 매수, 프로그램 매수 전략으로 신뢰도가 높아야 하는 전략이다. 평균 5.3점은 scorer의 해당 전략 케이스가 거의 매칭되지 않거나 감점 로직이 과도하게 작동하는 것을 의미한다.

**영향**
- S3/S5가 candidate_pool에 적재됨에도 불구하고 scoring 단계에서 100% 탈락
- ai-engine의 처리 부하가 높아지지만 실질 신호 기여 0

**확인 필요**
- `scorer.py`의 `S3_INST_FRGN`, `S5_PROG_FRGN` 케이스 로직
- 해당 전략의 입력 데이터 필드(inst_buy, frgn_buy, prog_buy 등) 실제 수신 여부
- `CLAUDE_THRESHOLDS` 임계값 vs 실제 평균 점수 괴리

## MEDIUM

### M-1. `stock_master`, `daily_indicators` 완전 미적재 (0건)

**증거**
```
stock_master    → 0건
daily_indicators → 0건
```

**분석**  
`stock_master`는 종목 기본 정보(업종, 시가총액, 상장일 등)를 담아야 하고, `daily_indicators`는 RSI, MACD, 볼린저밴드 등 기술지표 일배치 결과를 저장해야 한다. 둘 다 비어있으므로 해당 배치 잡이 한 번도 실행된 적 없다.

**영향**
- 종목 메타정보 기반 필터링 불가
- `daily_indicators`를 참조하는 스코어러 로직이 있다면 API 직접 호출로 대체되어 지연 증가

### M-2. `portfolio_config.enabled_strategies`와 실제 신호 발행 불일치

**증거**
```
portfolio_config.enabled_strategies:
  ["S1_GAP_OPEN", "S7_AUCTION", "S8_GOLDEN_CROSS", "S9_PULLBACK_SWING",
   "S10_NEW_HIGH", "S11_FRGN_CONT", "S12_CLOSING", "S13_BOX_BREAKOUT",
   "S14_OVERSOLD_BOUNCE", "S15_MOMENTUM_ALIGN"]

실제 신호 발행 전략 (trading_signals):
  S2_VI_PULLBACK (2건), S3_INST_FRGN (53건), S5_PROG_FRGN (40건),
  S6_THEME_LAGGARD (103건), S8_GOLDEN_CROSS, S9_PULLBACK_SWING, ...
```
S2, S3, S5, S6이 `enabled_strategies`에 없음에도 신호가 발행되고 있다.

**분석**  
`portfolio_config`가 실제 신호 발행 필터로 사용되지 않거나, api-orchestrator가 이 설정을 참조하지 않고 독립적으로 전략을 실행하는 것으로 추정된다.

**확인 필요**
- `TradingController` 또는 `StrategyService`에서 `portfolio_config.enabled_strategies` 참조 여부
- enabled_strategies 업데이트 필요 여부 판단
