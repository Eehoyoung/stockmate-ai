## 1. 근본 원인 분석 (Root Cause Analysis)

### 1-1. Python WS – 무한 재연결 루프

**증상**
```
[WS] 3.0초 후 재연결 (1번째) ← reconnect_count 가 항상 1로 리셋됨
→ 3초 간격으로 무한 재연결 반복
```

**원인**

| 항목 | 설명 |
|---|---|
| 환경 | 시스템 클럭이 UTC인 경우 `datetime.now(KST)` = UTC 시각 + 9h |
| 결과 | Python 프로세스 로그는 02:40 UTC를 표시하지만, 내부 KST 계산은 11:40 KST |
| 문제 | 11:40 KST는 07:30~15:35 범위 안 → `_is_market_hours()` = **True** |
| 파급 | `_wait_for_market_open()` 가 "이미 장 운영 중"으로 판단 → 즉시 반환 |
| 결과 | 서버가 10초 후 code=1000 "Bye" 로 종료해도 즉시 재연결 → 무한 루프 |

또한 mock API 서버는 구독 데이터가 없으면 **약 10초 후 code=1000 "Bye"** 로 연결을 종료하는 특성이 있어, 장 시간 중에도 빠른 재연결 루프가 발생할 수 있음.

**추가 문제**: `_wait_for_market_open()`이 정상 동작 하는 경우에도, `if not _is_market_hours(): continue` 이후 루프 상단에서 다시 호출될 때 이미 장 시간으로 판단되어 즉시 반환 → 재연결 루프.

### 1-2. Java – ka10029 Rate Limit (429/1700)

**증상**
```
ERROR [ctor-http-nio-5] KiwoomApiService: 4xx 오류 [ka10029] status=429
body={"return_msg":"허용된 요청 개수를 초과하였습니다[1700:...]"}
```

**원인 (복합)**

#### 원인 A: `retryWhen` 이 Rate Limiter를 우회

```java
// 기존 코드 – 버그 있음
public <T> T post(String apiId, ...) {
    WebClient.RequestBodySpec spec = createPostSpec(apiId, endpoint);
    // ↑ acquire() 는 여기서 1회만 호출
    T result = spec.bodyValue(body)
            .retrieve()
            ...
            .retryWhen(Retry.backoff(...))  // 재시도 시 spec 재사용 → acquire() 재호출 없음!
            .block(REQUEST_TIMEOUT);
}
```

`retryWhen`은 `spec.bodyValue(...).retrieve()...` 체인을 재실행하지만, `createPostSpec()` 내의 `rateLimiter.acquire()`는 **재호출하지 않음**. 따라서 재시도 시 Rate Limiter 토큰 없이 즉시 HTTP 요청 → Rate Limit 초과.

#### 원인 B: `CandidateService.getCandidates()` 가 장 시간 외에 ka10029 호출

```
/wsStart 수동 호출 (02:41 KST)
  → TradingController.startWs()
  → WebSocketSubscriptionManager.setupMarketHoursSubscription()
  → CandidateService.getAllCandidates()
  → CandidateService.getCandidates("001") + getCandidates("101")
  → Redis 캐시 없음 → fetchKa10029() × 2 호출!
```

DataQualityScheduler (60초 주기) 도 같은 경로로 `setupMarketHoursSubscription()`을 호출하여 ka10029를 반복 호출.

#### 원인 C: Rate Limiter 마진 부족

- 키움 제한: **초당 5회**
- 기존 설정: 초당 **4회** (250ms 간격)
- 동시 호출 + 재시도 burst 시 초과 가능

---

### 1-3. Java – Ping Executor 스케줄 누적

**증상**: 재연결마다 `onOpen`에서 새로운 Ping 스케줄 추가, 이전 스케줄 미취소.

**위험**: 다중 재연결 시 Ping 태스크 누적 → 불필요한 리소스 소모.

```

  장 시간 외 → _wait_for_market_open() 으로 다음 개장까지 대기

BYPASS_MARKET_HOURS=true (mock 테스트용):
  장 시간 무관하게 연결 시도
  지수 백오프: 3s → 6s → 12s → ... → 5분 (max)
  MAX_RECONNECTS 초과 → 5분 대기 후 카운터 리셋 → 무한 재시도
```

## 5. 알려진 제한 사항 (Known Limitations)

| 항목 | 내용 |
|---|---|
| Mock API 연결 지속 시간 | mock 서버는 실시간 데이터가 없어 ~10초 후 code=1000 으로 종료. 이는 mock 서버 특성으로 서비스 버그가 아님 |
| BYPASS 모드 무한 재시도 | `BYPASS_MARKET_HOURS=true` 는 mock 테스트 전용. 프로덕션은 반드시 `false` (기본값) |
| Rate Limiter 분산 환경 | 현재 Rate Limiter 는 단일 인스턴스 JVM 내 Semaphore. 다중 인스턴스 배포 시 Redis 기반 분산 Rate Limiter 필요 |
